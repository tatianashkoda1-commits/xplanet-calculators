# CLAUDE.md — инструкция для AI-ассистента

Этот файл — для Claude (или другого AI), который помогает разработчику интегрировать калькуляторы на сайт **xplanet.club**.

## Контекст

Это серия из 5 кулинарных калькуляторов, согласованных с автором проекта (шеф-повар, кондитер). UX и логика уже отлажены через несколько раундов правок — менять их без необходимости не нужно. Задача интеграции — встроить рабочие блоки на сайт, не сломав поведение.

## Архитектура

- 5 standalone HTML-файлов + 1 хаб (`index.html`)
- Каждый файл самодостаточный: `<style>` + `<script>` + разметка в одном HTML
- Внешние зависимости: только **Google Fonts** (Inter, Caveat)
- Никакого фреймворка, бандлера, npm — чистый HTML5/CSS3/ES2020+

## Точки интеграции

### 1. `cost.html` — словарь цен пользователя (КРИТИЧНО)

В демо-версии личный словарь хранится в `localStorage`. На проде должен жить в личном кабинете xplanet.club. В начале JS-блока есть отмеченная зона:

```js
// === ХРАНЕНИЕ ЦЕН — ЗОНА ИНТЕГРАЦИИ С ЛИЧНЫМ КАБИНЕТОМ XPLANET.CLUB ===
```

Четыре функции, которые надо подменить на API-вызовы:

- `loadPrices()` → `GET /api/user/prices` — список объектов
- `savePrice(item)` → `POST /api/user/prices` (или `PUT` если по `name` уже есть совпадение)
- `removePrice(name)` → `DELETE /api/user/prices/:name`
- `replaceAll(items)` → `PUT /api/user/prices` (массовая замена для импорта JSON)

**Структура объекта `item`:**

```ts
{
  name: string;           // короткое имя для матчинга ("мука", "молоко")
  unit: 'кг' | 'л' | 'шт'; // стандартная единица
  price: number;          // цена за стандартную единицу (всегда есть, рассчитывается из pack если pack задан)
  pack?: {                 // опциональные данные упаковки
    amount: number;
    unit: 'г' | 'кг' | 'мл' | 'л' | 'шт';
    price: number;
  } | null;
}
```

**Важно:** функции `loadPrices/savePrice/...` сейчас **синхронные**. При переходе на API сделать их `async` и обновить вызывающий код (`renderPantry`, `findInPantry`, обработчики кнопок) с `await`. Где это потребуется — в одном файле, всё рядом.

`portions.html`, `spoons.html`, `oven.html`, `form.html` — чистый клиент. Никаких API не требуют, можно встраивать как есть.

## Парсер рецептов (используется в `portions.html`, `form.html`, `cost.html`)

Логика парсинга строк рецепта одинаковая в трёх файлах. Если будете править — правьте согласованно.

**Регулярки:**

- Число (с точкой/запятой): `\d+(?:[.,]\d+)?`
- Диапазон: `(\d+(?:[.,]\d+)?)\s*[–—-]\s*(\d+(?:[.,]\d+)?)`
- Единицы: `^\s*(кг|мл|ст\.?\s*л\.?|ч\.?\s*л\.?|шт\.?|стакан\w*|г|л)(?=\s|$|[.,;:!?)])` — порядок альтернатив важен (длинные первые), lookahead вместо `\b` (см. грабли ниже)

**Что парсер не трогает:**

- Подзаголовки (`Тесто:`, `Начинка:`) — кончаются на `:`, нет цифр
- Альтернативы в круглых скобках в начале строки `(15 г свежих)` — не показываются в dropdown ингредиентов, но числа внутри пересчитываются как обычные
- «По вкусу», «щепотка», «на кончике ножа» — нет числа, оставляется как есть
- Концевые комментарии в скобках `(для хрусткости корочки)` — текст не трогается

## Грабли (известные, исправленные — НЕ ОТКАТЫВАТЬ)

1. **`type="number"` режет запятую.** Браузер сбрасывает `value="12,8"` в пустоту. Используется `type="text" inputmode="decimal" pattern="[0-9.,]*"` — даёт цифровую клавиатуру на мобиле + принимает запятую.
2. **`parseFloat("1,5")` возвращает 1, не 1.5.** Везде используется обёртка: `parseFloat(String(v).replace(',', '.'))` — функции `readNum()` в каждом файле.
3. **`\b` в JS regex не понимает кириллицу.** Для парсинга единиц используется lookahead `(?=\s|$|[.,;:!?)])` вместо `\b`.
4. **`\W` и `\w` тоже не понимают кириллицу.** `'мука'.split(/\W+/)` вернёт пустоту. Для извлечения слов на русском используется `s.match(/\p{L}+/gu)` (флаг `u`, Unicode property).
5. **GitHub Pages + кириллические имена файлов = 404.** Jekyll портит. Все имена латиницей, плюс `.nojekyll` в корне.
6. **Мягкое совпадение по падежам.** Логика в `cost.html` (`wordsMatch`):
   - 3-буквенные слова — только точное совпадение
   - 4-буквенные — общий префикс ≥ 3 (мука/муки)
   - 5+ — общий префикс ≥ 4 (сахар/сахара, изюм/изюма)
7. **Нормализация единиц** в парсере: `кг → г` (×1000), `л → мл` (×1000). Чтобы сравнивать «1 кг свинины» и «300 г» в одной шкале.
8. **«1 яйцо», «3 банана» — без явной единицы — считаются `kind='piece'`** (штучные). Если просто число + слово, парсер ставит `unit: 'шт'`. Не путать с «3 ст.л. масла» — там единица распознана как `ст.л.`, и `kind='custom'` (нужны граммы вручную).

## Стиль кода

- ES2020+, без транспиляции
- IIFE для каждого `<script>` — изолирует переменные
- Без `var`, только `const/let`
- Без сторонних библиотек, всё на vanilla
- Комментарии русские, кода — английские

## Что точно НЕ делать

- Не ломать UX-решения (переключатель «По порциям / По продукту», панель «Мой список цен» внутри cost-калькулятора, и т.д.) без согласования с автором.
- Не менять структуру объекта `item` в словаре цен — на эту структуру могут полагаться скрипты экспорта/импорта.
- Не упрощать парсер рецептов — он покрывает кейсы, которые автор реально использует (диапазоны, скобки, кириллица в падежах).

## Если что-то непонятно

Спрашивайте автора (через разработчика). Не угадывайте поведение — UX уже отлажен через тестирование на реальных рецептах коллектива.
